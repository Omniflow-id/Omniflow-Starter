# Failed Jobs Management ([/admin/queue/failed](/admin/queue/failed))

**Failed Jobs Management** 界面提供查看、分析和重试处理失败任务的功能工具。

## 快速操作

| 操作 | 方法 |
|------|------|
| 查看失败任务 | 访问 [/admin/queue/failed](/admin/queue/failed) |
| 重试单个任务 | 点击任务行的 **"Retry"** 按钮 |
| 批量重试任务 | 点击 **"Retry Failed Jobs"**（批量操作） |
| 查看错误详情 | 点击展开错误信息 |
| 查看所有任务 | 访问 [/admin/queue/jobs](/admin/queue/jobs) |

## 失败任务概览

### 什么是失败任务？

处理过程中出现错误的任务：
- **Worker 错误**：代码异常、崩溃
- **网络错误**：外部服务超时
- **数据错误**：无效负载、字段缺失
- **资源错误**：内存限制、数据库问题

### 任务何时被标记为失败

任务在以下情况被标记为"失败"：
1. Worker 抛出未处理的异常
2. 发生网络超时
3. 外部服务返回错误
4. 数据验证失败
5. 超过最大重试次数（3次重试）

## 任务信息显示

### 失败任务列表

| 列 | 说明 |
|----|------|
| **ID** | 任务唯一标识符 |
| **Queue** | 来源队列（email_queue、test_queue） |
| **Data** | 任务 JSON 负载（可展开） |
| **Attempts** | 重试次数（如：3/3 = 已达最大值） |
| **Error** | 详细错误信息（可展开） |
| **Created** | 初始任务创建时间 |
| **Started** | 最后一次处理尝试时间 |
| **Actions** | 重试按钮 |

### 错误信息显示

**可展开的错误详情：**
1. 点击错误文本进行展开
2. 查看完整错误信息和堆栈跟踪
3. 识别根本原因
4. 再次点击以折叠

**错误类型：**

| 错误模式 | 可能原因 |
|----------|----------|
| `Connection timeout` | 网络/外部服务问题 |
| `Validation error` | 无效的任务数据 |
| `Cannot read property` | 代码缺陷、数据字段缺失 |
| `ECONNREFUSED` | 外部服务宕机 |
| `SMTP error` | 邮件服务器问题 |
| `Database error` | 数据库连接/查询问题 |

## 重试功能

### 单个任务重试

**使用场景：**
- 根本原因修复后
- 调查单个任务
- 测试错误解决方案

**重试步骤：**
1. 识别需要重试的任务
2. 点击任务行的 **"Retry"** 按钮
3. 任务状态变为"Pending"
4. Worker 获取任务
5. 查看 [/admin/queue/jobs](/admin/queue/jobs) 获取结果

### 批量重试

**使用场景：**
- 系统性问题解决后（如：外部服务恢复上线）
- 网络中断恢复后
- 代码缺陷修复并部署后

**批量重试步骤：**
1. 点击页面顶部的 **"Retry Failed Jobs"** 按钮
2. 输入要重试的任务数量（如：10、50、100）
3. 确认操作
4. 系统优先重试最旧的失败任务
5. **结果**：任务从"Failed"移至"Pending"

**批量重试限制：**
- 每批最多 100 个任务（默认值）
- 防止队列过载
- 可以按顺序重试多批次

## 死信队列（DLQ）

### 什么是 DLQ？

**Dead Letter Queue** 存储满足以下条件的任务：
- 失败 3 次以上
- 超过最大重试次数
- 被标记为"无法恢复"

### DLQ 配置

| 设置 | 值 | 目的 |
|------|-----|------|
| **TTL** | 24 小时 | 1 天后自动删除 |
| **Max Attempts** | 3 | DLQ 阈值 |
| **Storage** | RabbitMQ DLQ | 与主队列分开存储 |

### DLQ 行为

**自动路由：**
1. 任务失败 3 次
2. 自动移至 DLQ
3. 在 [/admin/queue/failed](/admin/queue/failed) 仍可见
4. 可手动重试
5. 如果未重试，24 小时后过期

**24 小时 TTL 的好处：**
- 防止 DLQ 堆积
- 自动清理
- 强制在 24 小时内解决问题
- 减少存储需求

## 常见场景

### 场景 1：邮件服务中断

**问题：** 50 个邮件任务因 SMTP 超时而失败

**诊断：**
1. 访问 [/admin/queue/failed](/admin/queue/failed)
2. 查看错误信息："SMTP connection timeout"
3. 所有错误相同 → 外部服务问题

**解决方案：**
1. 确认邮件服务（SMTP）恢复上线
2. 测试单个任务：发送测试邮件
3. 如果成功：点击 **"Retry Failed Jobs"**
4. 输入 50 → 重试所有失败的邮件任务
5. 监控 [/admin/queue/jobs](/admin/queue/jobs) 直至完成

### 场景 2：错误的任务数据

**问题：** 单个任务因"Validation error"失败

**诊断：**
1. 点击任务行展开 **Data** 列
2. 查看 JSON 负载
3. 识别缺失/无效的字段
4. 错误信息："Missing required field: recipient"

**解决方案：**
1. **无法修复数据**：删除任务（如果功能可用）
2. **可以修复数据**：直接更新数据库（高级操作）
3. **预防**：创建任务前修复数据验证
4. **不要重试**：使用相同数据会再次失败

### 场景 3：代码缺陷修复

**问题：** Worker 代码存在缺陷导致 100 个任务失败

**解决流程：**
1. 开发者在 worker 代码中修复缺陷
2. 部署新版本 worker
3. 重启 worker 进程
4. 访问 [/admin/queue/failed](/admin/queue/failed)
5. **Retry Failed Jobs** → 输入 100
6. 在 [/admin/queue/jobs](/admin/queue/jobs) 监控处理进度
7. 验证成功完成

### 场景 4：达到 DLQ 阈值

**问题：** 任务已达到 3/3 次尝试，进入 DLQ

**理解：**
- 任务已失败 3 次
- 移至 Dead Letter Queue
- 将在 24 小时内过期
- 最后一次重试机会

**决策：**
1. **重试**：如果根本原因已修复
2. **忽略**：如果任务不再相关
3. **记录并删除**：用于审计追踪

### 场景 5：临时网络中断

**问题：** 网络中断期间 20 个任务失败

**快速恢复：**
1. 确认网络已恢复
2. 用 ping/curl 测试外部服务
3. 在 [/admin/queue](/admin/queue) 发送测试任务
4. 如果测试通过：**Retry Failed Jobs** → 20
5. 所有任务应成功完成

## 批量重试策略

### 重试前检查清单

批量重试前：
- [ ] 已识别根本原因
- [ ] 问题已解决/修复
- [ ] 测试任务成功
- [ ] Worker 运行中
- [ ] 网络/服务可用

### 批量重试大小

| 批量大小 | 适用场景 |
|----------|----------|
| 1-10 | 修复后测试、单个调查 |
| 11-50 | 小范围中断恢复、针对性重试 |
| 51-100 | 大范围中断、网络恢复 |
| 100+ | 分批执行、仔细监控 |

### 重试后监控

1. 导航至 [/admin/queue/jobs](/admin/queue/jobs)
2. 筛选 **"Processing"** → 查看活动任务
3. 筛选 **"Failed"** → 观察是否再次失败
4. 筛选 **"Completed"** → 验证成功
5. 如果再次失败，回顾错误模式

## 故障排除

| 问题 | 诊断 | 解决方案 |
|------|------|----------|
| 重试不工作 | Worker 未运行 | 检查 worker 状态，重启 workers |
| 任务立即再次失败 | 根本原因未修复 | 识别并修复底层问题 |
| 无法查看失败任务 | 数据库查询问题 | 检查 [/admin/cache/stats](/admin/cache/stats)，清除缓存 |
| 重试按钮禁用 | 任务正在重试中 | 等待当前重试完成 |
| DLQ 数量过高 | 系统性故障 | 回顾错误模式，修复根本原因 |
| DLQ TTL 已过期 | 任务自动删除 | 无法恢复，需要重新发送任务 |

## 错误分析

### 常见错误模式

**网络错误：**
```
ECONNREFUSED, ETIMEDOUT, ENOTFOUND
```
- **原因**：外部服务不可用
- **解决方案**：等待服务恢复，然后重试

**验证错误：**
```
Missing required field, Invalid email format
```
- **原因**：任务数据有问题
- **解决方案**：修复数据源，不要重试错误数据

**Worker 错误：**
```
TypeError, ReferenceError, Cannot read property
```
- **原因**：代码缺陷
- **解决方案**：修复代码，部署，然后重试

**资源错误：**
```
Out of memory, Database connection lost
```
- **原因**：资源耗尽
- **解决方案**：扩展资源，优化代码

### 错误调查步骤

1. **按错误信息分组** - 查找模式
2. **检查时间戳** - 识别中断窗口
3. **回顾任务数据** - 查找共同属性
4. **检查外部服务** - 验证连接性
5. **回顾 worker 日志** - 查找详细跟踪
6. **与活动日志关联** - 系统级视图

- 未修复原因
- 数据损坏/错误
- 过时的任务
- 调查之前

✅ **良好实践：**
- 监控 DLQ 数量
- 24 小时过期前调查
- 记录无法恢复的故障
- 为审计保留决策记录

❌ **避免：**
- 忽略 DLQ 任务
- 让任务过期
- 不经分析就重试
- 无错误文档
