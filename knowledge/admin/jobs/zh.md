# All Jobs 管理 (/admin/queue/jobs)

**All Jobs 管理** 界面提供了对系统中所有队列任务的全面可视性和控制功能，支持高级筛选和详细的任务信息。

## 快速操作

| 操作 | 方法 |
|------|------|
| 查看所有任务 | 访问 [/admin/queue/jobs](/admin/queue/jobs) |
| 按状态筛选 | 点击状态按钮（全部、待处理、处理中、已完成、失败） |
| 查看任务详情 | 点击行展开 JSON 数据 |
| 重试失败任务 | 访问 [/admin/queue/failed](/admin/queue/failed) |
| 翻页导航 | 使用底部分页控件 |

## 状态筛选

### 可用筛选器

| 筛选器 | 显示内容 | 徽章颜色 |
|--------|----------|----------|
| **全部** | 系统中的所有任务 | 混合色 |
| **待处理** | 等待处理的任务 | 黄色（警告） |
| **处理中** | 正在运行的任务 | 蓝色（信息） |
| **已完成** | 成功完成的任务 | 绿色（成功） |
| **失败** | 出错的任务 | 红色（危险） |

### 筛选方法

1. 访问 [/admin/queue/jobs](/admin/queue/jobs)
2. 点击按钮组中所需的状态按钮
3. 页面重新加载并显示筛选结果
4. 分页重置到第一页

## 任务信息显示

### 任务列表列

| 列 | 描述 | 详情 |
|----|------|------|
| **ID** | 任务唯一标识符 | 自增编号 |
| **Status** | 当前任务状态 | 带颜色徽章 |
| **Queue** | 目标队列名称 | `test_queue`、`email_queue` 等 |
| **Data** | 任务 JSON 数据 | 点击展开 |
| **Attempts** | 重试次数 | `当前/最大`（如：1/3） |
| **Created** | 任务创建时间 | 时间戳 |
| **Started** | 开始处理时间 | 未开始时为空 |
| **Completed** | 完成时间 | 未完成时为空 |
| **Error** | 失败详情 | 仅显示于失败任务 |

### 任务状态徽章

**待处理（黄色）：**
- 任务在队列中，等待 worker 接收
- 尚未发送到 RabbitMQ 或队列
- 处理前的正常状态

**处理中（蓝色）：**
- Worker 正在执行任务
- 数据库标记为"处理中"
- 应在几分钟内完成

**已完成（绿色）：**
- 任务成功完成
- Worker 返回成功
- 无错误记录

**失败（红色）：**
- 处理时发生错误
- 可能根据重试次数继续重试
- 错误详情显示在"Error"列

## JSON 数据查看器

### 查看任务数据

1. 点击表格中的任务行
2. **Data** 列展开显示格式化的 JSON
3. 查看完整的任务负载
4. 再次点击收起

```json

  "message": "Hello from admin panel",
  "timestamp": "2025-01-04T10:30:00.000Z",
  "triggeredBy": "admin@omniflow.id"
```

**邮件队列任务：**

### 数据损坏处理

如果 JSON 数据损坏或无效：
- **显示**：显示错误信息而非 JSON
- **可展开**：仍可点击查看原始数据
- **故障排除**：检查 worker 日志中的解析错误

## 任务生命周期

### 正常任务流程

```
1. 创建（待处理）→ 任务发送到队列
2. 处理中 → Worker 领取任务
3. 已完成 → 任务成功结束
```

**时间线：**
- **待处理**：数秒到数分钟（取决于队列深度）
- **处理中**：数秒到数分钟（取决于任务类型）
- **已完成**：永久状态

### 失败任务流程

```
1. 创建（待处理）
2. 处理中 → 发现错误
3. 失败 → 重试（如重试次数 < 最大值）
4. 处理中 → 重试失败
5. 失败（最终）→ 移至死信队列
```

**重试逻辑：**
- **最大重试次数**：3 次（默认）
- **重试延迟**：指数退避
- **死信队列阈值**：3 次及以上失败

## 分页

### 导航控件

- **上一页**：跳转到上一页
- **页码**：直接跳转到指定页面
- **下一页**：跳转到下一页
- **每页结果数**：默认每页 25 条任务

### 总记录数

底部显示格式：
```
显示 1-25 条，共 1,250 条任务
```

### 页面导航

1. 查看当前页的任务
2. 点击页码或上一页/下一页
3. 页面重新加载并显示新的一批任务
4. 筛选条件在整个分页过程中保持

## 常见场景

### 场景一：监控最新任务

**查看方法：**
1. 访问 [/admin/queue/jobs](/admin/queue/jobs)
2. 保持筛选器在"全部"
3. 任务按 ID 倒序排列（最新优先）
4. 检查最新任务的状态
5. 查找异常失败模式

### 场景二：调查处理中任务

**检查时机：**
- 任务卡在"处理中"状态
- 怀疑 worker 挂起
- 性能调查

**调查方法：**
1. 点击"处理中"筛选器
2. 检查 **Started** 时间戳
3. 如超过 10 分钟：可能已卡住
4. 查看 **Data** 列寻找线索
5. 检查 worker 日志
6. 考虑重启 workers

### 场景三：分析失败任务

**诊断流程：**
1. 点击"失败"筛选器
2. 查看 **Error** 错误信息
3. 寻找规律：
   - 相同错误重复 → 系统性问题
   - 不同错误 → 数据质量问题
   - 网络错误 → 外部服务故障
4. 点击 **Data** 查看任务负载
5. 访问 [/admin/queue/failed](/admin/queue/failed) 进行重试

### 场景四：审计任务历史

**用途：** 验证任务完成情况以满足合规要求

**流程：**
1. 筛选"已完成"
2. 翻页浏览
3. 检查 **Created** 和 **Completed** 时间戳
4. 计算处理时间
5. 导出到 [/admin/log](/admin/log) 的活动日志

### 场景五：清理旧任务

**维护任务：** 删除超过 30 天的已完成任务

**流程：**
1. 筛选"已完成"
2. 查看最早的任务（最后一页）
3. 记录任务 ID 以便删除
4. **管理员操作**：数据库清理（手动）
5. **替代方案**：实现自动清理定时任务

## 故障排除

| 问题 | 诊断 | 解决方案 |
|------|------|----------|
| 没有任务显示 | 队列为空或筛选问题 | 检查"全部"筛选器，验证任务是否存在 |
| 任务卡在待处理 | Workers 未运行 | 检查 worker 状态，必要时重启 |
| 任务卡在处理中 | Worker 挂起或长时间操作 | 检查 worker 日志，重启 workers |
| 失败任务数量多 | 系统性错误 | 检查 [/admin/queue/failed](/admin/queue/failed) 查看错误详情 |
| JSON 无法展开 | 浏览器 JavaScript 错误 | 刷新页面，检查控制台 |
| 分页不工作 | 缓存或会话问题 | 清除浏览器缓存，刷新页面 |

## 性能考量

### DataTable 实现

- **服务端处理**：支持处理数百万任务
- **分页**：每页仅加载 25 条任务
- **数据库索引**：status、queue、created_at 列快速查询
- **缓存集成**：结果缓存 2 分钟

### 缓存行为

**缓存键模式：**
```
datatable:jobs:{base64_query}
```

**TTL**：2 分钟

**失效时机：** 自动失效于：
- 任务状态变更
- 新任务创建
- 失败任务重试

### 查看缓存统计

在 [/admin/cache/stats](/admin/cache/stats) 查看缓存性能：
- 任务查询的命中率
- 内存使用情况
- 响应时间

### 数据库结构

**jobs 表：**
```sql
- id (主键，自增)
- queue (字符串，已索引)
- data (JSON 数据)
- status (枚举：pending, processing, completed, failed)
- attempts (整数)
- max_attempts (整数，默认 3)
- error (文本，可空)
- available_at (时间戳)
- started_at (时间戳，可空)
- completed_at (时间戳，可空)
- created_at, updated_at
```
